# 腾讯云精简 Supabase 共存部署设计

## 文档信息

| 项目 | 内容 |
|---|---|
| 日期 | 2026-07-17 |
| 状态 | 用户已确认，可进入实施 |
| 应用 | 学业闭环 / 2026 暑期家教作业系统 |
| 公网应用域名 | `summerwork.ilelezhan.cn` |
| 公网 API 域名 | `summerwork-api.ilelezhan.cn` |
| 部署模式 | 腾讯云共享 CVM 上的精简 Supabase，与现有项目隔离共存 |
| 核心约束 | 不停止、不修改、不迁移服务器上的任何现有项目 |

本文件不记录服务器密码、数据库密码、API 密钥、SSH 私钥或真实家庭信息。

## 1. 目标与边界

### 1.1 目标

1. 在中国大陆腾讯云服务器上运行现有学业闭环网页、Supabase Auth、PostgreSQL、REST、Realtime 和 API 网关。
2. 保留现有 Next/Vinext 应用、16 份数据库迁移、200 个任务块、173 项作业本体和三角色权限模型。
3. 让家长、分科家教和孩子通过两个 HTTPS 域名稳定访问，不再依赖境外 Supabase 网络。
4. 对现有容器、数据库、Nginx 站点和后台服务保持零停机、零数据变更。
5. 提供可独立停止、可保留数据回滚、可校验备份和可复现部署的运行边界。

### 1.2 非目标

- 不清理、升级或重构服务器上的现有项目。
- 不复用现有 PostgreSQL、Redis、Node 进程、Docker 网络或数据卷。
- 不部署 Supabase Storage、imgproxy、Edge Functions、Analytics、Logflare、Vector 或公网 Studio。
- 不在本批次启用 UFW、执行系统升级或重启服务器。
- 不发送微信、短信、作业提醒邮件或账号验证邮件。
- 不保存孩子作业照片，不代替学校平台提交作业。

## 2. 已核验服务器基线

只读体检确认：

- Ubuntu 24.04 LTS，腾讯云 CVM，2 核、约 8GB 内存、80GB 系统盘。
- 根分区剩余约 62GB，现有内存可用量约 6GB，已配置约 2GB Swap。
- Docker、Docker Compose、Node.js 与 Nginx 已安装并运行。
- 服务器已经运行多套 Node、Python、PostgreSQL、Redis、MQTT 和 Docker 项目。
- 公网 `80`、`443` 及多组应用端口已被现有服务使用；Nginx 负责现有域名与默认站点。
- 两个新域名已解析到该服务器，但尚无独立 Nginx 虚拟主机，因此当前落入默认站点。
- UFW 当前未启用；本批次不改变该状态，公网边界由腾讯云安全组和 Nginx 控制。

官方完整 Supabase 的推荐资源高于当前 CPU 配置，因此本设计只部署业务必需组件，并设置独立资源上限和停止阈值。

## 3. 方案比较与选择

| 方案 | 优点 | 风险/代价 | 结论 |
|---|---|---|---|
| A. 精简 Supabase 共存 | 保留现有代码和 RLS；成本最低；国内访问 | 共享 2 核 CPU，需要严格资源限制 | **已选择** |
| B. 完整 Supabase 共存 | 功能齐全 | 组件多、资源高，可能影响现有项目 | 不采用 |
| C. 独立 4 核 8GB 服务器 | 隔离最彻底 | 增加服务器费用 | 本机资源不足时的回退方案 |

若精简栈在稳态测试中越过资源停止阈值，不降级现有服务、不强行上线，转而建议方案 C。

## 4. 目标架构

```text
家长 / 家教 / 孩子
        |
        | HTTPS 443
        v
summerwork.ilelezhan.cn
        |
        | 新增独立 Nginx server block
        v
127.0.0.1:3180
学业闭环应用容器

浏览器与服务端
        |
        | HTTPS + WebSocket 443
        v
summerwork-api.ilelezhan.cn
        |
        | 新增独立 Nginx server block
        v
127.0.0.1:8180
Supabase API 网关
   |-- Auth
   |-- REST / PostgREST
   |-- Realtime
   `-- 独立 PostgreSQL
```

### 4.1 目录和命名隔离

- 根目录：`/srv/summerwork`
- 应用源码：`/srv/summerwork/app`
- 部署配置：`/srv/summerwork/deploy`
- 备份目录：`/srv/summerwork/backups`
- Docker Compose 项目名：`summerwork`
- 容器、网络、数据卷和镜像标签统一使用 `summerwork_` 前缀。
- 不使用现有项目目录，不执行全局 Docker 清理，不删除非本项目镜像或卷。

### 4.2 精简服务范围

保留：

- PostgreSQL：权威数据、RLS、RPC、迁移和审计事件。
- Auth：邮箱＋密码账号和会话。
- PostgREST：浏览器与服务端数据 API。
- Realtime：任务、通知和状态实时刷新。
- API 网关：统一 API、Auth 和 WebSocket 入口。
- 学业闭环应用：响应式三角色网页。

不部署：Storage、imgproxy、Edge Runtime、Analytics、Logflare、Vector、公开 Studio。数据库管理与迁移通过 SSH 和受控命令执行。

## 5. 网络与端口

### 5.1 腾讯云安全组需要开放

| 端口 | 协议 | 来源 | 用途 |
|---:|---|---|---|
| 80 | TCP | 公网 | ACME 证书校验和跳转 HTTPS |
| 443 | TCP | 公网 | 两个域名的 HTTPS、Auth、REST 与 Realtime WebSocket |
| 22 | TCP | 仅管理员固定出口 IP，条件允许时 | SSH 运维 |

不需要开放任何 UDP 端口。

### 5.2 仅服务器本机监听

| 地址 | 用途 | 是否开放安全组 |
|---|---|---|
| `127.0.0.1:3180` | 学业闭环应用 | 否 |
| `127.0.0.1:8180` | Supabase API 网关 | 否 |

### 5.3 仅 Docker 内网监听

PostgreSQL、Auth、PostgREST 和 Realtime 端口只存在于 `summerwork` Docker 网络，不绑定 `0.0.0.0`，不开放腾讯云安全组。数据库 `5432` 不对公网或局域网暴露。

## 6. Nginx 与证书隔离

1. 修改前记录 `nginx -T`、现有站点列表和全部现有网站响应基线。
2. 备份 `/etc/nginx` 配置元数据和拟接触的文件；不复制或输出 TLS 私钥内容。
3. 只新增两个命名明确的站点配置，不编辑现有三个站点文件。
4. 证书使用独立 ACME Webroot 方式申请，不允许 Certbot 自动改写现有站点。
5. 新配置先执行 `nginx -t`；失败时不 reload，并删除未启用的新链接。
6. 配置验证成功后使用平滑 reload，不重启 Nginx，不中断现有连接。
7. API 站点显式传递 WebSocket Upgrade/Connection 头，保证 Realtime 可用。
8. HTTPS 稳定后再启用新站点自身的 HSTS，不改变其他域名策略。

## 7. 账号与权限

### 7.1 注册方式

- 使用邮箱＋密码注册，Supabase Auth 自动确认邮箱，不发送验证邮件。
- 密码最低 12 位。
- 注册成功本身不授予家庭、孩子或科目数据权限。
- 家长在系统内生成邀请链接后，通过系统外的私密渠道单独转发。

### 7.2 单家庭启动门

上线前创建唯一家长管理员和首个家庭空间。数据库增加单家庭启动保护：

- 系统尚无家庭时，首个受控家长账号可以创建家庭。
- 首个家庭创建后，其他普通账号不能再创建第二个家庭。
- 未受邀请账号即使注册成功，也不能读取现有家庭、孩子、任务或知识证据。

### 7.3 邀请与撤权

- 邀请令牌只保存摘要，原令牌仅在创建时返回。
- 邀请 7 天有效、只能使用一次，并绑定目标邮箱、角色、孩子和科目。
- 接受邀请时当前登录邮箱必须匹配。
- 家教仅获得指定孩子＋指定科目权限。
- 撤销家教后，旧登录会话立即失去本科访问；历史署名与审计记录保留。

### 7.4 无验证邮箱的残余风险

账号邮箱所有权不经过邮件服务验证，因此邀请链接本身是重要凭证。若链接和目标邮箱同时泄露，第三方可能抢先注册并接受邀请。缓解措施是：限时、一次使用、邮箱绑定、Nginx 登录限速、站内审计，以及要求家长只通过一对一渠道发送邀请。

## 8. 密钥与服务器访问

- 所有数据库密码、JWT 密钥、Publishable Key、Secret Key 和备份密钥使用密码学安全随机值生成。
- 服务器 `.env` 与备份配置仅管理员可读，权限设置为 `0600`。
- Secret Key 不写入前端包、不提交 Git、不出现在日志和验收报告中。
- 应用运行时只获得公开 URL 和 Publishable Key；管理脚本单独读取 Secret Key。
- 先增加并验证独立 SSH 密钥，再要求用户通过腾讯云控制台轮换已在聊天中暴露的密码。
- 本批次不擅自关闭 SSH 密码登录，因为该全局设置可能影响其他运维流程；如需关闭，另行确认。

## 9. 数据迁移与初始化

1. 在新的独立 PostgreSQL 中执行仓库 `supabase/migrations/0001` 至最新迁移。
2. 任何一份迁移失败即停止；不跳过、不手工修改部分表后继续。
3. 同步权威 `summer-2026-family-tutoring` 目录。
4. 自动复核：200 个任务块、173 项作业本体、6 科、23 个课程日。
5. 复核排除：7 月 5 日、英语、政治、历史、地理均不进入计划。
6. 复核语文全部进入考背课程、自主或机动单元。
7. 正式家庭数据在合成账号验收完成并清理后创建，避免测试数据与真实数据混合。

## 10. 资源保护与停止条件

- 新应用和 Supabase 组件合计设置独立 CPU/内存资源限制，目标总内存上限约 3.5GB。
- 公网启用前至少进行 15 分钟稳态观察。
- 稳态期间服务器可用内存不得持续低于 2GB。
- 不允许出现 OOM、持续 Swap 增长、容器重启循环或现有网站响应失败。
- 2 核服务器上若 5 分钟负载持续接近或超过 CPU 容量，停止新栈并分析，不牺牲现有项目资源。
- 部署前后比较现有容器健康、systemd 服务、监听端口和网站 HTTP 状态。
- 任一保护条件失败：不配置公网 Nginx，停止 `summerwork` 容器但保留新数据卷用于分析。

## 11. 备份与恢复

- 每日凌晨以低 CPU/IO 优先级执行 PostgreSQL 逻辑备份。
- 每份备份压缩并生成 SHA-256 校验值，文件权限 `0600`。
- 默认保留 14 天，清理脚本仅匹配 `summerwork` 备份目录。
- 部署验收包含一次恢复到临时数据库的演练，证明备份可读取且迁移结构完整。
- Docker 停止或回滚不使用 `down -v`，数据库卷和备份始终保留。
- 后续建议增加腾讯云磁盘快照或异机加密副本；该动作涉及云控制台和额外成本，不在本批次自动执行。

## 12. 分阶段部署

1. **本地部署资产**：新增最小 Compose、网关、Nginx 模板、备份/验证脚本和部署说明；先通过本地测试并提交 Git。
2. **服务器检查点**：记录基线与回滚命令，创建独立目录，不启动公网服务。
3. **数据库样本**：启动最小数据库和 API 栈，执行迁移与200条计划导入。
4. **本机验收**：只通过 `127.0.0.1` 检查 Auth、REST、Realtime 和应用。
5. **四账号合成验收**：家长、数学家教、物理家教和孩子完成登录、邀请、跨科拒绝、学习、批改、订正、掌握、提交和撤权链路；完成后自动清理。
6. **资源静置**：至少15分钟资源与现有服务对比。
7. **证书和域名**：申请两个证书，新增独立站点，`nginx -t` 后平滑 reload。
8. **公网验收**：手机/桌面 HTTPS、登录、刷新、WebSocket、站内通知和断线恢复。
9. **备份恢复**：生成首份备份，在临时数据库验证恢复后删除临时实例。
10. **访问加固**：验证 SSH 密钥，用户轮换暴露密码，留下运行和回滚手册。

## 13. 验收标准

### 13.1 不影响现有项目

- 部署前后的现有 Docker 容器集合与健康状态一致。
- 现有 systemd 服务仍为运行状态。
- 现有 Nginx 配置语法、域名响应和 TLS 不退化。
- 不占用已有公网/本机端口，不修改已有数据卷和数据库。
- 不执行全局 Docker prune、系统升级、服务器重启或防火墙切换。

### 13.2 新系统功能

- 两个新域名证书有效且强制 HTTPS。
- Auth 注册、登录、退出和会话刷新有效。
- REST 与数据库 RPC 正常；Realtime WebSocket 可建立并在状态变化后刷新。
- 200 个任务块、173 项作业本体及排除规则全部通过自动复核。
- 本科家教可读写本科，跨科家教被拒绝，孩子不能批改或提交确认。
- 闭环只有在首做、批改、订正/复做、掌握和全部必需提交节点完成后成立。
- 撤销邀请/授权、重复点击、版本冲突和软删除恢复均符合既有集成测试。

### 13.3 运行可靠性

- 新栈无容器重启循环，无 OOM，资源未越过停止条件。
- 每日备份可生成校验值，并至少完成一次临时恢复演练。
- 回滚命令只停用新域名和 `summerwork` Compose 项目，不删除数据卷。

## 14. 回滚方案

按影响从外到内执行：

1. 停用两个新 Nginx 站点链接，执行 `nginx -t` 后平滑 reload。
2. 执行仅针对 `summerwork` 项目的 Compose stop/down，不附带 `-v`。
3. 保留源码目录、密钥文件、数据库卷、备份和证书供诊断。
4. 对照部署前基线复核现有网站、容器、服务和端口。
5. 如需重新部署，从最后一个本地 Git 检查点和保留的数据卷继续；不修改其他项目。

## 15. 用户需要完成的云侧操作

- 确认两个域名 A 记录持续指向该服务器。
- 腾讯云安全组允许公网 TCP 80、443。
- SSH 22 仅保留必要来源，优先限制到管理员固定出口 IP。
- 部署完成并确认 SSH 密钥可用后，在腾讯云控制台轮换已暴露的服务器密码。
- 不需要开放 3180、8180、5432 或任何 Supabase 内部端口。

## 16. 设计结论

在“不影响任何现有项目”的硬约束下，采用精简 Supabase、独立 Compose 项目、loopback 端口、独立 Nginx 虚拟主机和分阶段门禁。该方案复用现有 Auth/Postgres/RLS/Realtime 代码，同时把共享服务器风险限制在可观测、可停止和可保留数据回滚的范围内。若资源门禁失败，唯一允许的扩展方案是迁移到独立服务器，而不是降低现有项目资源或跳过验收。
